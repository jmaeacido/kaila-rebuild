# ADR-0010 — Server authority, auditability, idempotency, and concurrency

**Decision date:** 2026-07-10  
**Applies to:** New KAILA platform and later migration work

**Status:** Accepted

## Context

Job transitions, offer acceptance, timers, retries, mobile reconnects, and future realtime delivery can produce duplicate requests and race conditions.

## Decision

Security-sensitive and lifecycle-changing operations are server-authoritative and use:

- database transactions;
- explicit authorization policies;
- optimistic concurrency or equivalent version checks;
- idempotency keys for retryable commands;
- append-only domain/audit events;
- durable timestamps generated by the server;
- transactional outbox or an equivalent reliable event-publication pattern when external delivery is involved.

Clients may request a command but may not submit the resulting authoritative state. APIs return conflict responses when a resource version or precondition is stale.

## Consequences

- Duplicate network requests do not create duplicate offers, acceptances, completions, reviews, or transitions.
- State history can be reconstructed without relying solely on current-row values.
- Realtime/push delivery is downstream of committed database changes.
- Foundation code must make the secure path the easiest path for later phases.
